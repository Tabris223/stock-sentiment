### 三、成本和限流问题怎么解决？

**结论：成本可忽略，限流有余量，真正的问题是数据质量和反馈闭环。**

#### 3.1 成本评估（精确版）

**GLM-4-Flash 定价**：
- 输入：0.001 元/千 token
- 输出：0.001 元/千 token

**单条新闻成本估算**：
```
输入 token 数（约）：
- 系统提示词：300 token
- 新闻标题：50 token
- 新闻摘要：200 token
- 总计：约 550 token

输出 token 数（约）：
- 情绪 + 置信度 + 摘要 + 关键点：150 token
- 新增维度（事件类型/影响时间/信号强度）：50 token
- 总计：约 200 token

单条成本 = (550 × 0.001 + 200 × 0.001) / 1000 = 0.00075 元
```

**月成本估算**：
| 场景 | 每天新闻数 | 每月成本 |
|------|----------|---------|
| 当前（9 只股票） | 50 条 | 1.1 元 |
| 扩展（50 只股票） | 300 条 | 6.8 元 |
| 激进（100 只股票） | 600 条 | 13.5 元 |

**结论：成本不是问题。即使扩展 10 倍，月成本也不到 20 元。**

#### 3.2 限流评估（精确版）

**GLM-4-Flash 限流**：10 RPM（每分钟 10 次请求）

**实际需求**：
| 场景 | 每分钟最大需求 | 限流余量 |
|------|--------------|---------|
| 定时抓取（每 5 分钟） | 10 条/5 分钟 = 2 RPM | 80% 余量 |
| 实时抓取（每分钟） | 50 条/分钟 | **超限** |

**结论**：
- **定时抓取**：10 RPM 足够
- **实时抓取**：需要批量拼接（已实现）或申请提升限流

**如果需要实时抓取**：
1. 使用批量拼接（10 条新闻一次请求，减少 90% 调用）
2. 申请提升限流到 60 RPM（联系智谱 AI）
3. 使用消息队列削峰填谷

#### 3.3 真正的问题：数据质量和反馈闭环

**成本和限流不是问题，真正的问题是：**

1. **数据质量**：RSS 抓什么就分析什么，没有噪音过滤
   - 重复新闻（同一事件多家媒体报道）
   - 无关新闻（行业动态但不影响个股）
   - 低质量内容（标题党、广告）

2. **反馈闭环缺失**：没有用户反馈，无法知道分析是否准确
   - 用户看到的结果是对是错？
   - 哪些错误是用户可以容忍的？
   - 哪些错误会导致用户流失？

3. **Benchmark 缺失**：没有标注数据集，无法量化改进
   - 我们不知道准确率到底是多少
   - 优化后不知道提升了多少

**我的建议**：
- **优先级 P0**：建立 100-200 条标注数据集（我在 Phase 1 会做）
- **优先级 P1**：增加"反馈"按钮，让用户标记错误
- **优先级 P1**：增加去重和噪音过滤逻辑

## Round 1 原始发言（保留）

### 一、词典法 vs LLM：差距比你想象的大

我看了代码。词典法（SentimentAnalyzer）和 LLM（LLMAnalyzer）的差距不是"60% vs 86%"这种数字差异，而是**根本性的能力差异**。

#### 1.1 词典法做不到的事

**无法理解语境和否定**

```
"招商轮船并未受到国际贸易摩擦的负面影响"
```

词典法会抓到"负面影响"里的"负面"，判断为利空。实际上这是**利好**（没受影响）。

**无法区分"现象"和"原因"**

```
"招商轮船股价下跌，但分析师认为基本面健康，建议逢低买入"
```

词典法会抓到"下跌"，判断为负面。但分析师的判断是正面，这才是有价值的信息。

**无法处理多主题**

```
"招商轮船三季度营收增长20%，但现金流承压，同时宣布重大资产重组计划"
```

这是三个不同维度：业绩（正面）、现金流（负面）、重组（不确定）。词典法会简单地加权平均，输出一个无意义的"中性"。

**无法识别"标题党"和"情绪渲染"**

财经媒体喜欢用夸张标题吸引点击，但内容可能是中性甚至相反的。词典法只看标题关键词，会被误导。

#### 1.2 LLM 能做到的事（但需要调试）

LLM 理论上可以：
- 区分"股价下跌"（市场噪音）vs "基本面恶化"（真实风险）
- 识别新闻主体（这新闻是关于招商轮船，还是关于整个航运行业？）
- 判断时效性和重要性（这是旧闻，还是突发事件？）
- 评估信息来源可靠性（官方公告 vs 媒体转述 vs 社交传闻）

**但前提是 Prompt 要写好，模型要调过。**

我们现在的 Prompt 模板（sentiment_analysis.md）写得不错，有边界情况处理、置信度标准。但**没有在生产环境验证过**。

#### 1.3 真实差距估算

我没法给出精确数字，因为**词典法没有 benchmark**。但我可以基于经验估算：

| 能力维度 | 词典法 | LLM（当前） | LLM（优化后） |
|---------|--------|------------|--------------|
| 简单正面/负面判断 | 60-70% | 80-85% | 85-90% |
| 否定句处理 | 30-40% | 75-80% | 85-90% |
| 多主题拆解 | 0% | 60-70% | 75-85% |
| 标题党识别 | 0% | 50-60% | 70-80% |
| 事件分类（业绩/政策/人事/行业） | 0% | 70-80% | 80-90% |

**结论：词典法只适合做"有没有情绪"的粗筛，不适合做投资参考。**

**我不是在抱怨。我是在说：这个项目的"产品需求"从来没有和"分析能力"对齐过。**

#### 4.3 舆情分析不是"调个 API 就行"

Werner 你提到"GLM-4-Flash 够不够"。这个问题本身就暴露了对舆情分析的误解。

**模型只是 20% 的工作。80% 是：**

1. **数据质量控制**：噪音过滤、去重、来源可信度
2. **Prompt 工程**：让模型输出结构化、可解析的结果
3. **后处理**：置信度校准、边界情况处理
4. **反馈闭环**：用户反馈 → 标注 → 微调

我们现在的项目里：
- 数据质量控制：没有（RSS 抓什么就分析什么）
- Prompt 工程：有（但没在生产环境验证）
- 后处理：没有（解析失败就用默认值）
- 反馈闭环：没有

**这不是"GLM-4-Flash 够不够"的问题。这是"有没有把 LLM 当成真正的分析系统来建设"的问题。**

---

### 五、最后一个问题：我们真的需要"情绪分析"吗？

Marty 提到一个用户场景：

> 老王想知道：今天有没有影响股价的大事？有的话，是利好还是利空？要不要操作？

这个场景里，"情绪"只是第三层。第一层是**"有没有大事"**。

如果 90% 的新闻都是噪音（常规公告、媒体转载、无关行业动态），那用户真正需要的是**"信号检测"**，不是"情绪分析"。

**我的建议：在优化"情绪分析"之前，先定义"什么算大事"。**

| 事件类型 | 算不算"大事" | 触发条件 |
|---------|------------|---------|
| 业绩公告 | ✅ | 每季度 |
| 重大资产重组 | ✅ | 事件驱动 |
| 高管变动 | ⚠️ 视级别 | CEO/董事长 |
| 行业政策 | ⚠️ 视影响 | 直接影响公司 |
| 股价波动 | ❌ | 市场噪音 |
| 媒体报道 | ❌ | 通常无新信息 |

**如果我们连"大事"都定义不清楚，"情绪分析"做得再准也是优化了错误的问题。**

总结三句话：
1. **词典法不能用**，差距是根本性的，不是量化差异
2. **当前维度不够**，需要从"情绪分析"升级到"信息结构化"
3. **没人定义过目标**，我到现在不知道"成功"的标准是什么

#### 2.1 数据层：SQLite → PostgreSQL

**当前决策**：SQLite 够用，>1GB 时迁移

**触发条件清单**：

| 触发条件 | 阈值 | 谁负责检测 | 检测频率 | 决策 |
|---------|------|----------|---------|------|
| 数据库大小 | > 1GB | Linus | 每周 | 迁移 PostgreSQL |
| 并发写入冲突 | 锁等待 > 5s | Linus | 实时告警 | 迁移 PostgreSQL |

| 数据保留策略 | 需要保留 > 6 个月 | Marty | 产品决策时 | 迁移 PostgreSQL |

- **10 人**：10 人同时访问时，SQLite 的并发读性能开始成为瓶颈

**执行建议**：

1. Linus 在监控面板加一个"数据库大小"指标（每周自动上报）

3. 提前准备好 PostgreSQL 迁移脚本（不要等到触发条件才写）

### 三、决策机制建议

Round 1 我说"团队往三个方向跑了"。这是因为**没有决策机制**。

#### 3.1 谁决策？

**原则：决策权在"对结果负责的人"。**

| 决策类型 | 谁决策 | 谁执行 | 谁审核 |
|---------|--------|--------|--------|
| **产品方向** | PM（Marty） | 全员 | CTO（Werner） |
| **技术选型** | 架构师（Martin） | 后端/前端 | CTO（Werner） |
| **架构演进** | 架构师（Martin） | 后端（Linus） | CTO（Werner） |
| **优先级** | CTO（Werner） | 全员 | - |
| **质量标准** | 舆情分析（Selene） | 后端（Linus） | 架构师（Martin） |

**具体规则**：

1. **产品方向由 PM 决策**
   - Marty 决定"用户是谁"、"核心价值是什么"
   - 但 CTO 可以否决（如果技术成本过高）

2. **技术选型由架构师决策**
   - 我决定"用什么数据库"、"用什么框架"
   - 但 CTO 可以否决（如果违反公司技术战略）

3. **架构演进由架构师提出，CTO 批准**
   - 我提出"什么时候迁移 PostgreSQL"
   - CTO 批准后，Linus 执行

4. **优先级由 CTO 决策**
   - Werner 决定"先做什么，后做什么"
   - 这是项目管理的根本

#### 3.2 怎么决策？

**原则：决策要"明确、书面、有触发条件"。**

**决策流程**：

```
1. 提出问题 → 2. 收集信息 → 3. 定义触发条件 → 4. 做出决策 → 5. 书面记录 → 6. 执行
```

**示例**：

| 决策 | 问题 | 收集的信息 | 触发条件 | 决策 | 记录位置 |
|------|------|----------|---------|------|---------|
| 数据库迁移 | SQLite 够不够？ | Linus 容量估算（100 只股票 × 1 年 ≈ 4-21GB） | 数据 > 1GB | 迁移 PostgreSQL | 架构决策文档 |
| 前端简化 | Dashboard 要不要？ | 无用户验证 | 打开率 < 20% | 砍掉 | 本文档 |
| LLM 切换 | 词典法还能用吗？ | Selene 分析（无法处理否定句、多主题） | - | 直接切换 | 本文档 |

**关键规则**：

1. **每个架构决策必须有触发条件**
   - 没有"以后再说"，只有"当 X 发生时，我们做 Y"
   - 触发条件必须可量化、可检测

2. **决策必须书面记录**
   - 不写下来的决策 = 没决策
   - 记录在本项目的 `docs/architecture-decisions/` 目录

3. **决策后不要反复推翻**
   - 除非有新的数据证明决策错误
   - "我觉得"不是推翻决策的理由

#### 3.3 防止往三个方向跑

**Round 1 的问题**：后端在做 LLM，产品在写技术方案，前端在画 Dashboard，没人对齐。

**防止机制**：

1. **每周架构同步会（30 分钟）**
   - 参与人：架构师 + 后端 + 前端 + PM
   - 内容：
     - 本周做了什么架构决策？
     - 下周计划做什么？
     - 有没有跨团队的依赖？
   - 输出：架构同步会议纪要

2. **重大决策必须通知全员**
   - 什么是"重大决策"？影响 > 1 个模块的决策
   - 通知方式：飞书群 + 文档记录
   - 异议期：24 小时内可以提出异议

3. **代码审查包含"架构审查"**
   - PR 不只是审查代码质量，还要审查"是否符合架构方向"
   - 如果 PR 引入了新的复杂性，必须说明理由

4. **季度架构审视**
   - 每 3 个月，架构师回顾：
     - 哪些决策执行了？哪些没执行？
     - 架构熵增了多少？（代码行数、模块数、依赖数）
     - 下季度需要做什么架构调整？

### 五、防止架构熵增的具体措施

Round 1 我说"前端 255MB 是架构熵增的典型例子"。  
怎么防止？

#### 5.1 代码审查清单（每次 PR 必查）

| 检查项 | 问题 | 不合格的标准 |
|--------|------|------------|
| **新增依赖** | "这个依赖真的需要吗？" | 依赖体积 > 500KB |
| **新增模块** | "这个模块能不能合并？" | 模块间耦合 < 30% |
| **新增配置** | "这个配置能不能简化？" | 配置项 > 10 个 |
| **新增接口** | "这个接口能不能删除？" | 接口使用率 < 20% |
| **代码行数** | "这段代码有没有重复？" | 重复代码 > 50 行 |

**执行人**：架构师（我）review 每个 PR 的架构影响

#### 5.2 季度架构审视（每 3 个月）

**审视指标**：

| 指标 | 当前值 | 警戒值 | 超过警戒值的行动 |
|------|--------|--------|----------------|
| **总代码行数** | 2,112 行（后端） | +50% | 重构简化 |
| **前端组件数** | 50 个 | +30 个 | 删除未使用组件 |
| **依赖数量** | 253MB node_modules | +100MB | 清理未使用依赖 |
| **模块数量** | 6 个模块 | +3 个 | 合并相似模块 |

**审视流程**：

1. 我统计当前值，对比上一季度
2. 如果超过警戒值，提出重构建议
3. CTO 决定是否执行重构
4. 执行后记录在 `docs/architecture-review/`

#### 5.3 架构决策记录（ADR）

**每个重大决策都要记录**：

```
# ADR-001: 选择 SQLite 作为初始数据库

## 决策
使用 SQLite 作为 MVP 阶段的数据库

## 理由
1. 轻量级，无需额外部署
2. 数据量预计 <1GB，SQLite 性能足够
3. 迁移到 PostgreSQL 成本低（SQL 兼容）

## 触发条件
- 数据 > 1GB → 迁移 PostgreSQL
- 用户 > 10 人 → 迁移 PostgreSQL

## 影响
- 后端：Linus 负责监控数据大小
- 架构：Martin 负责准备迁移脚本

## 决策人
架构师 Martin，2026-03-05
```

**存放位置**：`docs/architecture-decisions/`

#### 5.4 "YAGNI 原则"检查

**每次新增功能前，问三个问题**：

1. **"用户真的需要这个吗？"**
   - 有用户反馈吗？没有 → 不做
   - 有数据支持吗？没有 → 不做

2. **"能不能用更简单的方式？"**
   - 有没有现成的解决方案？
   - 能不能复用现有的代码？

3. **"这个功能 3 个月后还需要吗？"**
   - 如果不确定，先不做
   - 等用户反馈明确后再做

**执行人**：PM（Marty）+ 架构师（Martin）在需求评审时检查

## 共识与结论

### 一、形成的共识

1. **用户价值优先于技术完成度** — 我们的技术优化（API限流、批处理、监控）做得不错，但没有验证过用户是否需要这些。7天内，用数据说话，不是用代码行数说话。

2. **词典法立即废弃** — 不是"并行运行观察"，是直接删除。Selene的分析已经证明：词典法和LLM的差距是"能不能用"的问题，不是"60%还是86%"的问题。让用户用会误导的工具，违背产品初衷。

3. **架构演进需要触发条件** — 好的架构不是设计出来的，是演化出来的。但演化需要方向和触发条件。"当X发生时，做Y"比"以后再说"有用一万倍。

4. **决策机制必须明确** — 产品方向PM决策，技术选型架构师决策，优先级CTO决策。决策要书面记录，要有触发条件，不能反复推翻。

5. **砍掉比堆砌更难** — 好产品是砍出来的。Dashboard 7天生存测试，不达标就砍，不留"以后再说"。
